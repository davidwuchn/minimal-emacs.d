## Failure Mode Hypotheses

Without captured `example reasons`, the failure type is inferred from the `prompt` label. The following are the most probable root causes, ranked by evidence weight:

### H1: Self-referential prompt construction (HIGH confidence)
`gptel-tools-agent-prompt-build.el` is in the failure set. Modules that *build* prompts for the agent are failing when *fed* prompts by the agent. This circular dependency is a classic source of prompt degradation.

```elisp
;; Hypothesized anti-pattern in gptel-tools-agent-prompt-build.el
(defun gptel-build-prompt (context)
  (let ((prompt-template (gptel-fetch-template context)))
    ;; The template itself was generated by a failing prompt:
    (format prompt-template ...)))
```

**Fix:** Decouple template generation from template instantiation. Templates should be versioned and committed, not regenerated on every invocation.

### H2: Shared prompt template drift (MEDIUM confidence)
The `auto-workflow-strategic` and `auto-workflow-projects` modules share an identical trend timestamp (`2026-06-04T013900Z-dfe2`). This strongly suggests a single prompt template was used to generate both, and the template is producing output the auto-workflow monitor rejects.

**Fix:** Diff the prompts used for these two modules. Likely they should diverge but are inheriting from a shared (broken) base.

### H3: Catastrophically low keep-rate (HIGH confidence)
The two modules with explicit keep-rates:

| Category | Keep-rate | Reject rate | Implication |
|---|---|---|---|
| `:programming` | 3.5% | 96.5% | The agent is essentially non-functional for programming tasks |
| `:agentic` | 4.3% | 95.7% | The agent is essentially non-functional for agentic tasks |

A keep-rate below 5% is not "the prompt is suboptimal" — it is "the prompt is broken at a structural level." Likely causes:
- Prompt is requesting output in a format the model cannot produce
- Prompt has conflicting or impossible constraints
- Prompt references modules/context that don't exist in the model's training
- Token limit is being exceeded silently

**Fix:** Before tuning, verify the prompt is *answerable*. Manually execute the top 3 prompts and confirm a human can produce the expected output.

### H4: Missing diagnostic capture (MEDIUM confidence)
All 7 records show `**Example reasons:** none`. The monitoring agent is detecting failures but not capturing *why* they failed. This makes the pattern undebuggable from the mistake records alone.

**Fix:** Extend the mistake-capture hook to record at least the first 200 chars of the rejected output or a one-line error message.

## Actionable Patterns

### Pattern 1: Isolate the prompt-build module
**Symptom:** `gptel-tools-agent-prompt-build.el` appears in the failure set.
**Action:** Audit this module first. It is a likely root cause for the entire `tools-agent` cluster.
**Verification:** Regenerate one other `tools-agent` module *without* going through `prompt-build`; if the failure count drops, the hypothesis is confirmed.

### Pattern 2: Batch-correlated failures
**Symptom:** Two or more modules share an identical trend timestamp.
**Action:** Suspect a shared prompt template. Diff the inputs and outputs across the batch.
**Example from data:** `strategic` and `projects` both end at `2026-06-04T013900Z-dfe2`.

### Pattern 3: Reject-rate-driven triage
**Symptom:** Keep-rate < 10% for a category.
**Action:** Pause generation in that category. Do not iterate on the prompt — restart with a hand-written example. The prompt is not "tunable"; it is broken.
**Threshold:** Keep-rate < 5% → halt and rewrite. Keep-rate 5–20% → iterate with new examples. Keep-rate > 20% → normal prompt tuning.

### Pattern 4: Missing example reasons
**Symptom:** Mistake records have `**Example reasons:** none`.
**Action:** Patch the mistake-capture pipeline to record rejection reason. This is a prerequisite for any meaningful pattern analysis.

## Code Template: Hand-Written Prompt Skeleton

When a category's keep-rate is below 5%, replace the auto-generated prompt with this skeleton:

```elisp
(defun gptel-generate--MODULE-NAME (input)
  "Generate output for MODULE-NAME from INPUT.
INPUT is a plist with keys: :context, :constraints, :examples.
Returns a string or signals `gptel-generation-failed'."
  (let* ((system-prompt
          (concat
           "You are an Emacs Lisp generator. "
           "Produce ONLY valid Emacs Lisp code. "
           "Do not include explanations, markdown, or commentary.\n\n"
           "Required defuns: <list>\n"
           "Required exports: <list>\n"
           "Forbidden patterns: <list>\n"))
         (examples
          (gptel-fetch-examples :module 'MODULE-NAME :n 3))
         (user-prompt
          (format "Input: %s\nExamples:\n%s\nOutput:"
                  input
                  (mapconcat #'identity examples "\n---\n"))))
    (gptel-request
     :system system-prompt
     :user user-prompt
     :on-success (lambda (out) (with-temp-file target out))
     :on-failure (lambda (err) (signal 'gptel-generation-failed err)))))
```
