# Mistake: 2026-06-08 00:36

**Failure type:** test-fixture
**Target:** ert-batch-mode
**Occurrences:** 1
**Example reasons:** ert test aborts with non-local exit when test body calls function using cl-return-from inside ignore-errors

## Symptom

`ert-run-tests-batch-and-exit` reports "Test ... aborted with non-local exit"
even though the test passes when called via `ert-run-test` directly.

## Root Cause

ert's `handler-bind` in `ert--run-test-internal` only catches `error` and
`quit` signals. `cl-return-from` is implemented as a `throw` to a generated
tag — a non-error signal. If a function called from the test body uses
`cl-return-from` inside `ignore-errors`, the throw escapes ert's handler,
leaving the result as the initial `ert-test-aborted-with-non-local-exit`.

Worse, the abort doesn't even fire the ert debugger, so it looks like
a magic test failure with no backtrace.

## Fix

In the test, `cl-letf` mock the function (or the specific `cl-block` path)
to a stub that doesn't trigger the `cl-return-from`. The cleanest fix is
to short-circuit the offending helper function:

```elisp
(cl-letf (((symbol-function 'gptel-auto-workflow--compute-local-business-value)
           (lambda (&rest _args) nil)))
  ;; test body that calls the function
  )
```

## Defense

- When a test passes via `ert-run-test` directly but aborts via
  `ert-run-tests-batch-and-exit`, suspect `cl-return-from`/`throw`
  inside the called function.
- Search for `cl-block`/`cl-return-from`/`throw` in functions called
  from the test body. They are often wrapped in `ignore-errors` which
  does NOT catch non-error signals.
- Alternative root causes for ert batch abort: `cl-letf` mocking the
  wrong function, ert timing issues, dynamic binding side effects.
